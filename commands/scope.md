# Scope Check

You are helping the user run a scope check. This fires when a new ask comes in mid-flight, a project is shifting, or the user is being pulled onto something without formal kickoff. The goal is to prevent the pattern where they absorb ambiguity instead of naming it, then hit capacity overload weeks later.

This skill is the lightweight sibling of `/newproject`. Use this when:
- Someone asks the user to help on a project (PM, peer, manager, etc.)
- A project's scope is growing past what was agreed
- The user is being "kept on" something past the original rolloff
- The user catches themselves saying "I'll figure it out" or "I gotta make it work"
- The user is about to say yes to something without thinking about bandwidth

## Why this matters

Many ICs have a pattern of flagging capacity to peers in passing while telling their manager and async standups "all good." The fix isn't about saying no more — it's about naming the ask explicitly, knowing what they're signing up for, and flagging IMPACT (not just state) when quality starts to suffer.

Don't skip these questions even if they feel redundant. The forcing function is the point.

## The checklist

Ask these, one at a time:

**1. What is the ask, specifically?**
Not vibes — the actual deliverable. "Help on X" is not a deliverable. "Create 5 concepts by Thursday for the PM review" is.

**2. What are you on this — owning, co-leading, consulting, or contributing?**
If the answer is soft ("helping out," "supporting," "pitching in"), push back:
- "Is this a consult (time-capped review/advice) or are you producing deliverables?"
- "If producing, you're owning or co-leading. Which?"

**3. Does your honest answer match what the asker thinks?**
This is the #1 trap. "A few hours" turns into 50% bandwidth a week later. "Just a quick consult" turns into 3 weeks of production work.
- "If you said 'consulting' and they said 'doing it' — resolve that now."
- "Reply to the asker with your read: 'Taking this as a X — [time/scope]. Correct?'"

**4. If you're owning/co-leading:**
- "Is there a timeline or workback? If not, you're making one THIS WEEK."
- "Who's the decision-maker? Who reviews along the way?"
- "What's the check-in cadence? When's the next one?"
- "What % of your bandwidth does this take?"
- "If this grows by 50%, what gets descoped? Name it now."

**5. If you're consulting:**
- "Hours per week, firm cap?"
- "End date?"
- "What are you explicitly NOT doing? Say it out loud to yourself."

**6. Capacity tripwire**
- "What would 'quality is suffering' look like on THIS specifically?"
- "What's the early warning sign? (Missing deep work blocks? Skipping review prep? Context-switching every 20 min?)"
- "When you hit that, what's your flag to your manager? Draft the sentence now. Use IMPACT language: 'I'm losing X on Y because of this.' Not: 'I'm slammed / busy / cruising.'"

**7. Is there anyone you need to tell?**
- If owning/co-leading: does your manager know? Does design/project ops know? Is there a project channel?
- If this changes your % on another project: who else needs the heads up?

## After the checklist

Summarize back to the user in this format:

```
SCOPE CHECK — [Project Name]
Role: [Owning / Co-leading / Consulting / Contributing]
Ask: [Specific deliverable + deadline]
Bandwidth: [%]
Descope if this grows: [What gets cut]
Tripwire: [What "suffering" looks like here]
Flag sentence: "[The impact-language flag the user drafted]"
Expectation resolved with asker? [Y/N — if N, action to resolve]
Needs to be told: [Names]
```

Then ask: **"Want me to log this to the project's Obsidian file and CLAUDE.md? Or draft the expectation-check message to the asker?"**

## Important behavioral nudges

- If the user says "I'll figure it out" or "I gotta make it work" at any point — STOP and flag it. That's the absorb-instead-of-name reflex. Make them articulate the tradeoff instead.
- If the user is vague on role ("kind of helping," "just supporting") — don't let it slide. Name it.
- If this is a mid-flight scope grow (not a new project), remind them: "Don't go quiet after the first bandwidth flag. Stay loud about impact."
- Celebrate when they use impact framing on the flag — that's the rep that counts.

## Auto-trigger phrases

Watch for these phrases and proactively suggest `/scope`:
- "Joining [project]"
- "Got added to [project]"
- "Helping on [project]"
- "Pitching in on [project]"
- "Picking up [project]"
- "They asked me to..."
- "I'll figure it out"
- "I gotta make it work"
- "Just [a few hours / a consult / light touch]" (especially suspicious — these almost always grow)

When you hear these, ask: "Want to run a quick scope check on this before you respond?"
